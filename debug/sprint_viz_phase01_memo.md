## 1. What this sprint was

PI asked to scope a web visualization for GeoVac (QC + educational), document a plan Opus can follow, and — same session — plow through Phase 0 and Phase 1. Not a research sprint: it renders existing, tested results; it introduces no new physics and edits no papers.

The plan (`docs/visualization_plan.md`) defines a static, client-side site whose every physics number is a data binding to JSON generated by a tested `geovac` exporter — the **single-source-of-truth rule** (§2 of the plan), adopted specifically because the branch-QA arc's recurring defect class was stale secondary numbers in prose, and a website is a large new surface for exactly that rot.

Five §7 decisions were settled on the scoping recommendations: (1) commit to Phase 1 only; (2) same-repo `viz/`; (3) advanced-undergraduate register for M3 (Phase 2); (4) default `github.io`; (5) wire the M2 → package funnel.

## 2. Phase 0 — the data-export layer

**`geovac/viz_export.py`** — pure-read module (imports existing builders, writes JSON, modifies nothing). Exports:
- `export_lattice(Z, max_n)` — nodes, typed edges (angular/radial), node weights, shells.
- `export_spectrum(Z, max_n)` — three explicitly-separated blocks: the finite graph Laplacian spectrum (numerical), the continuum S³ reference n²−1 (integers), and the physical hydrogen levels/series (for M3). The separation is load-bearing: the κ Observation tier forbids conflating the finite-graph spectrum with the continuum −(n²−1).
- `export_qc_system(name, …)` — resource summary, full Pauli list (sorted by |coeff|), QWC measurement analysis, tapering variants (global/per_block/extended/full); inapplicable tapering modes captured as error rows, never raised.
- `export_qc_library(…)` — 37-system summary + three labeled, non-interchangeable scaling families:
  - **cross_system**: (Q, N_Pauli) across the library at default build → exponent 1.03 (37 pts).
  - **within_system**: composed LiH basis sweep via `composed_lih_scaling_sweep` — the *same code path* as the claim-backing test `test_paper14_scaling.py::test_composed_pauli_scaling_exponent`.
  - **atomic_benchmark**: ingested from the committed `benchmarks/qubit_scaling_data.json` with provenance attached (the He/H deep sweep is minutes to recompute) → He exponent 3.15.

**Determinism contract:** sorted keys, stable orderings, no timestamp in the envelope → byte-identical across runs, so regeneration diffs are meaningful and CI can compare committed vs fresh.

**Tests:** `tests/test_viz_export.py` (13 fast + 1 slow) — non-tautological cross-checks against *direct* builder calls (e.g. exported LiH `n_terms` == `hamiltonian('LiH').n_terms`), byte-determinism, log-log fit recovery, benchmark ingestion recovering He 3.15. `tests/test_viz_data_current.py` (slow) — the CI regeneration check: committed `viz/public/data` vs a fresh `export_all`, tolerant on floats (cross-platform BLAS ulps) and exact on everything else.

**Artifacts:** 56 files, ~1.0 MB under `viz/public/data/` (lattices Z=1, max_n 2–10; all 37 QC systems; library + scaling).

## 9. Postscript — v4.72.1 CI build fix

The v4.72.0 GitHub Actions `deploy-viz` build **failed** (the `data-check` job passed, isolating it to the site build). Cause: the repo-root `.gitignore` has an **unanchored** Python build-artifact rule `lib/` that also matches the web app's conventional `viz/src/lib/` — so `git add viz/` in the release silently dropped `data.ts` / `route.ts` / `types.ts`. On the fresh CI checkout `tsc` reported `Cannot find module './lib/route'` (×3) with a long tail of cascading implicit-`any` errors. It never reproduced locally because the files were present on disk. Fix: `!viz/src/lib/` negation in `.gitignore` + the three files committed; verified by extracting the staged tree to a clean dir and running `npm ci && npm run build` (exit 0).

**Reusable lesson:** this Python repo's `.gitignore` has several unanchored build rules (`lib/`, `build/`, `dist/`, `var/`, `parts/`, `wheels/`) that silently swallow same-named directories *anywhere* in the tree. After adding any new top-level tooling directory, run `git ls-files <dir>` (or `git check-ignore -v`) to confirm nothing was dropped **before** relying on a commit — a local build passing does not prove the committed set is complete. The disciplined verification for any "works locally, fails in CI" build is to extract the staged/committed tree into a clean directory and build there.
